## Scope

- `tintinweb/pi-subagents` at `8405e556acc8ab2c0fcfa02021f871365695f776` (upstream source for installed `@tintinweb/pi-subagents` v0.13.0)

## Executive summary

`gentle-pi` has a GOOD orchestration contract, but most of the Orchestrator -> skill matching/injection behavior is prompt policy, not hard runtime enforcement. The strongest reusable parts are:
- thin-parent orchestration,
- exact-path skill injection,
- artifact references instead of artifact bodies,
- SDD phase envelopes + read-back gatekeeping,
- package-owned worker precedence for bounded multi-file writes.

The weakest parts for `pi-workflow` are:
- no code-level skill matcher/injector in `gentle-pi` itself,
- a shallow generated skill registry that can diverge from `pi-subagents` recursive skill loading,
- mixed `skill_resolution` contracts,
- model-routing coupling to `subagents.json` fields that `@tintinweb/pi-subagents` v0.13.0 does not appear to consume.

## Evidence-backed source facts

### 1) Skill discovery / indexing / source of truth

1. `gentle-pi` generates a local registry at `.atl/skill-registry.md` plus a fingerprint cache at `.atl/.skill-registry.cache.json`; the generator explicitly says `SKILL.md` is the source of truth and the registry is an index for delegators, not a summary. It scans many project/user roots, parses only frontmatter `name` and `description`, excludes `_shared`, `skill-registry`, and `sdd-*`, and deduplicates by skill name with project paths preferred over user paths. ([extensions/skill-registry.ts#L17-L31](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L17-L31), [extensions/skill-registry.ts#L54-L95](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L54-L95), [extensions/skill-registry.ts#L128-L228](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L128-L228), [extensions/skill-registry.ts#L265-L301](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L265-L301))
2. The registry refresh is lifecycle-managed: startup refresh, forced `/skill-registry:refresh`, fingerprint cache hit short-circuit, file watchers, `.atl/` gitignore insertion, skip flags/env, duplicate-extension suppression, and quarantine of a stale legacy project-local registry extension. ([extensions/skill-registry.ts#L251-L320](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L251-L320), [extensions/skill-registry.ts#L367-L407](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L367-L407), [extensions/skill-registry.ts#L425-L469](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L425-L469), [extensions/skill-registry.ts#L471-L613](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L471-L613))
3. The registry is intentionally shallow per root: `findSkillFiles()` only checks direct child directories for `SKILL.md`, not recursive nested skills; the test suite locks that behavior. ([extensions/skill-registry.ts#L97-L126](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L97-L126), [tests/skill-registry.test.ts#L117-L145](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/tests/skill-registry.test.ts#L117-L145))
4. `pi-subagents` does not use a registry. Its own `skills:` frontmatter preloads skill content directly by name from ordered roots, with recursive BFS for `SKILL.md` directories, and injects full skill content into the child system prompt. ([src/skill-loader.ts#L1-L102](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/skill-loader.ts#L1-L102), [src/agent-runner.ts#L321-L333](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L321-L333), [src/prompts.ts#L46-L56](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L46-L56))

### 2) How the Orchestrator matches skills and injects them

1. In `gentle-pi`, this is primarily an orchestrator prompt contract, not code. The parent is instructed to read `.atl/skill-registry.md`, match task context/target files against `Trigger / description`, and pass exact `Path` values under `## Skills to load before work`; the child should read those exact files and should not rediscover the registry. ([assets/orchestrator.md#L91-L99](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L91-L99), [assets/orchestrator-skills.md#L5-L18](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-skills.md#L5-L18))
2. `gentle-pi` also defines `skill_resolution` feedback semantics for orchestration health: `paths-injected`, `fallback-registry`, `fallback-path`, `none`. The parent is told to treat fallback as an orchestration gap and inject exact indexed paths next time. ([assets/orchestrator-skills.md#L19-L26](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-skills.md#L19-L26))
3. SDD phase agents repeat that contract: use parent-injected exact paths first; fallback self-loading is allowed only as degraded self-healing. ([assets/agents/sdd-design.md#L17-L21](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-design.md#L17-L21), [assets/agents/sdd-apply.md#L15-L23](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-apply.md#L15-L23))
4. But the bounded writer contract is stricter and incompatible: `gentle-ai-worker` allows only `paths-injected | paths-invalid | none` and explicitly forbids fallback registry/path values; the package test suite locks that stricter contract. ([assets/agents/gentle-ai-worker.md#L72-L101](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/gentle-ai-worker.md#L72-L101), [tests/package-manifest.test.ts#L231-L307](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/tests/package-manifest.test.ts#L231-L307))
5. In `pi-subagents`, exact-path injection is not a first-class mechanism. Named `skills:` are resolved by the extension and their full contents are appended to the child system prompt. ([src/custom-agents.ts#L55-L77](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/custom-agents.ts#L55-L77), [src/agent-runner.ts#L321-L333](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L321-L333), [src/prompts.ts#L46-L56](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L46-L56))

### 3) How subagent definitions / models / thinking / tool scopes / prompt modes / context inheritance are configured

1. `gentle-pi` ships agent definitions as markdown assets and installs them globally under the Pi agent home (`agents/`, `chains/`, `gentle-ai/support/`). That is asset installation, not a runtime spawn engine. ([lib/sdd-preflight.ts#L154-L179](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/lib/sdd-preflight.ts#L154-L179), [README.md#L229-L237](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/README.md#L229-L237))
2. Packaged `gentle-pi` agents mostly specify `name`, `description`, and YAML-list `tools`; they do not usually pin `prompt_mode`, `inherit_context`, `run_in_background`, `model`, or `thinking` in the asset frontmatter. That means the effective defaults come from the subagent runtime. ([assets/agents/sdd-design.md#L1-L13](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-design.md#L1-L13), [assets/agents/sdd-apply.md#L1-L12](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-apply.md#L1-L12), [tests/package-manifest.test.ts#L202-L218](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/tests/package-manifest.test.ts#L202-L218))
3. `gentle-pi`'s `/gentle:models` layer stores global routing in `~/.pi/gentle-ai/models.json`, still reads a legacy project-local `.pi/gentle-ai/models.json`, normalizes `model` + `thinking`, and applies them by updating non-builtin agent frontmatter plus writing a `model_profiles` object into `subagents.json`. ([extensions/gentle-ai.ts#L653-L740](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L653-L740), [extensions/gentle-ai.ts#L865-L890](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L865-L890), [extensions/gentle-ai.ts#L1244-L1320](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L1244-L1320))
4. In `@tintinweb/pi-subagents` v0.13.0, custom/frontmatter agent config is authoritative over call-site params for `model`, `thinking`, `max_turns`, `inherit_context`, `run_in_background`, `isolated`, and `isolation`. Omitted `prompt_mode` defaults to `replace`; omitted `inherit_context` and `run_in_background` default to `false`. ([src/custom-agents.ts#L51-L77](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/custom-agents.ts#L51-L77), [src/invocation-config.ts#L13-L35](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/invocation-config.ts#L13-L35), [README.md#L203-L229](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L203-L229))
5. `pi-subagents` also owns tool and extension scoping. Built-in tool names are derived from Pi's own tool factories; `extensions:` controls what loads, `tools:` controls what surfaces, `ext:` narrows extension tools, and `disallowed_tools` removes access even if something was otherwise available. ([src/agent-types.ts#L12-L22](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-types.ts#L12-L22), [src/custom-agents.ts#L53-L64](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/custom-agents.ts#L53-L64), [src/agent-runner.ts#L377-L563](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L377-L563), [README.md#L211-L260](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L211-L260))
6. Prompt mode behavior is owned by `pi-subagents`: `append` uses the parent prompt plus a sub-agent bridge and keeps a cacheable parent-prefix; `replace` uses a standalone child prompt. ([src/prompts.ts#L15-L28](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L15-L28), [src/prompts.ts#L32-L93](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L32-L93))
7. Context inheritance is also owned by `pi-subagents`: when enabled, the runtime builds a text summary from prior user/assistant messages and compaction summaries, skipping tool results, then prepends that to the child task prompt. ([src/context.ts#L15-L58](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/context.ts#L15-L58), [src/agent-runner.ts#L658-L676](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L658-L676))
8. Default `pi-subagents` agent types are embedded runtime definitions: `general-purpose` is an `append` parent twin, while `Explore` and `Plan` are `replace` read-only agents; `Explore` is pinned to `anthropic/claude-haiku-4-5`. ([src/default-agents.ts#L11-L70](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/default-agents.ts#L11-L70), [README.md#L152-L160](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L152-L160))

### 4) How parent context stays thin / artifact references vs full content

1. `gentle-pi` explicitly defines the parent as a thin coordinator and lazily splits detailed orchestration domains into separate assets (`orchestrator-delegation.md`, `orchestrator-memory.md`, `orchestrator-skills.md`, `sdd-orchestrator-workflow.md`). The rendered orchestrator prompt is cached. ([assets/orchestrator.md#L9-L13](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L9-L13), [assets/orchestrator.md#L69-L99](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L69-L99), [extensions/gentle-ai.ts#L118-L140](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L118-L140))
2. For non-SDD memory use, `gentle-pi` says the parent selects memory context and the child writes back; the child should not go search memory for context on its own. ([assets/orchestrator.md#L79-L89](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L79-L89))
3. For SDD, `gentle-pi` is explicit that the parent passes artifact references, not artifact bodies; each phase child reads its required inputs from Engram/OpenSpec itself and persists its own output. ([assets/orchestrator-memory.md#L5-L23](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-memory.md#L5-L23), [assets/agents/sdd-design.md#L28-L40](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-design.md#L28-L40), [assets/agents/sdd-apply.md#L25-L41](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-apply.md#L25-L41))
4. `pi-subagents` defaults to fresh child context because `inherit_context` defaults to `false`; that aligns with a thin orchestrator when the parent simply passes a self-contained task prompt. ([src/invocation-config.ts#L26-L35](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/invocation-config.ts#L26-L35), [README.md#L223-L226](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L223-L226))

### 5) Launch/result contracts, gatekeeping, deduplication, recovery

1. `gentle-pi` defines an SDD phase result envelope: `status`, `executive_summary`, `artifacts`, `next_recommended`, `risks`, `skill_resolution`. In auto mode, the parent must gate on contract conformance, artifact existence, no hallucinated references, no scope drift, and routing coherence; it may rerun a failed phase once with corrective feedback. ([assets/sdd-orchestrator-workflow.md#L120-L155](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/sdd-orchestrator-workflow.md#L120-L155))
2. `gentle-pi` also hardens delegation routing: prefer `subagent_run` when available, use `mode: "task"` when continuation depends on the result, prefer the package-owned `gentle-ai-worker` for bounded multi-file writes, and fall back to Pi `Agent` only when needed. ([assets/orchestrator-delegation.md#L45-L66](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-delegation.md#L45-L66), [assets/orchestrator-delegation.md#L107-L120](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-delegation.md#L107-L120))
3. There is real deduplication/recovery around the skill-registry extension itself: duplicate package/project extension loads are skipped, stale generated project-local registry extensions are quarantined, and startup can self-heal by forcing a regeneration. ([extensions/skill-registry.ts#L451-L469](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L451-L469), [extensions/skill-registry.ts#L551-L583](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L551-L583), [tests/skill-registry.test.ts#L181-L215](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/tests/skill-registry.test.ts#L181-L215))
4. I did not find a code-level `subagent launch dedupe` / `task fingerprint` mechanism in `gentle-pi`; that pattern is described in orchestration prompt policy elsewhere, but not implemented in this package source. The concrete implemented dedupe I found is duplicate skill-registry extension load suppression. ([extensions/skill-registry.ts#L451-L469](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L451-L469))
5. `pi-subagents` provides the actual runtime launch/recovery features: run/resume/steer, session creation, tool/activity tracking, graceful max-turn wrap-up, and context inheritance. ([README.md#L13-L33](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L13-L33), [src/agent-runner.ts#L572-L676](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L572-L676))

### 6) Which behavior comes from gentle-pi vs pi-subagents vs Pi itself

1. `gentle-pi` provides packaged prompt assets, SDD agent/chain/support assets, the skill-registry generator, model-routing writer, persona/harness prompt, and safety/orchestration policy. It does not declare a dependency on `pi-subagents`; it only recommends a companion package in README. ([package.json#L43-L61](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/package.json#L43-L61), [README.md#L68-L78](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/README.md#L68-L78))
2. `pi-subagents` provides the actual subagent runtime: agent registry, frontmatter parsing, prompt construction, tool/extension/skill loading, model resolution, context inheritance, memory injection, background execution, steering, and session lifecycle. ([src/custom-agents.ts#L11-L79](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/custom-agents.ts#L11-L79), [src/prompts.ts#L15-L93](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L15-L93), [src/agent-runner.ts#L300-L676](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L300-L676))
3. Pi core (`@earendil-works/pi-coding-agent`) provides the built-in tool factories, `DefaultResourceLoader`, `SettingsManager`, `SessionManager`, and `createAgentSession`; `pi-subagents` builds on those rather than reimplementing them. ([src/agent-types.ts#L8-L22](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-types.ts#L8-L22), [src/agent-runner.ts#L8-L18](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/agent-runner.ts#L8-L18))

## pi-workflow recommendations

### Copy

1. Copy the thin-parent contract: keep the parent as coordinator, keep always-on prompt small, and lazy-load detailed orchestration surfaces only when needed. `gentle-pi`'s split between `orchestrator.md` and lazy detail assets is a good pattern. ([assets/orchestrator.md#L9-L13](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L9-L13), [extensions/gentle-ai.ts#L118-L140](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L118-L140))
2. Copy the registry-as-index idea, not registry-as-summary. Exact `SKILL.md` path injection is the right abstraction for preserving author intent and avoiding prompt drift. ([extensions/skill-registry.ts#L265-L301](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L265-L301), [tests/skill-registry.test.ts#L33-L52](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/tests/skill-registry.test.ts#L33-L52))
3. Copy the artifact-reference rule: parent passes Engram/OpenSpec references, children fetch their own required artifacts and persist their own outputs. That fits `pi-workflow`'s Engram-first persistence decision VERY well. ## Proposed responsibility map for pi-workflow

### Orchestrator
- Own task classification, delegation thresholding, Engram context selection, artifact-reference passing, and gatekeeping.
- Execute a real skill resolver against the registry/index plus task metadata.
- Inject exact `SKILL.md` paths and fail loudly when required paths are missing/unreadable.
- Never pass full prior conversation by default.

### Agent frontmatter
- Own stable per-agent runtime defaults: `model`, `thinking`, `prompt_mode`, `tools`, `extensions`, `disallowed_tools`, `inherit_context`.
- Default all package-owned workers/reviewers/SDD executors to fresh context.
- Keep model pinning here when possible because `@tintinweb/pi-subagents` definitely consumes it.

### Skill registry / resolver
- Registry: generated index only, recursive, project-over-user precedence, exact paths, invalidation on startup/manual refresh/watch.
- Resolver: real code, not prose, that maps task shape + touched paths + intent to concrete skill paths.
- Output: exact path list only; never compact summaries.

### pi-subagents extension
- Treat as the execution substrate: sessions, tool scoping, prompt building, extension loading, steering/resume/background, optional context inheritance.
- Do not ask it to decide workflow policy; that belongs above it.
- Assume only documented/implemented knobs (`model`, `thinking`, `prompt_mode`, `tools`, `extensions`, `disallowed_tools`, `inherit_context`, `isolation`).

### Engram
- Source of durable project context, user constraints, SDD state, and artifact references.
- Parent reads/selects context; child phases read required artifacts by topic key and persist outputs.
- Review/gatekeeper reads artifacts back from Engram to verify reality, not summaries.

## Main risks to carry forward

1. **Registry/runtime drift**: `gentle-pi` indexes skills shallowly, `pi-subagents` loads skills recursively. Copying both patterns unchanged will create false negatives. ([extensions/skill-registry.ts#L97-L126](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L97-L126), [src/skill-loader.ts#L62-L101](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/skill-loader.ts#L62-L101))
2. **Prompt-only enforcement risk**: in `gentle-pi`, exact skill injection is mostly a prompt rule, not a runtime guard. The parent can silently fail to do it. ([assets/orchestrator.md#L91-L99](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator.md#L91-L99), [assets/orchestrator-skills.md#L5-L26](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/orchestrator-skills.md#L5-L26))
3. **Contract skew**: SDD agents allow fallback enums, `gentle-ai-worker` forbids them. That is exactly the kind of split-brain contract you do NOT want in `pi-workflow`. ([assets/agents/sdd-design.md#L17-L21](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/sdd-design.md#L17-L21), [assets/agents/gentle-ai-worker.md#L91-L99](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/assets/agents/gentle-ai-worker.md#L91-L99))
4. **Model-routing coupling risk**: `gentle-pi` writes `model_profiles` into `subagents.json`, but `@tintinweb/pi-subagents` v0.13.0 source/docs show frontmatter and operational settings handling, not `model_profiles` consumption. Builtin-agent routing therefore looks risky or no-op under this runtime. ([extensions/gentle-ai.ts#L1244-L1320](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/gentle-ai.ts#L1244-L1320), [src/settings.ts#L1-L117](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/settings.ts#L1-L117), [src/custom-agents.ts#L55-L77](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/custom-agents.ts#L55-L77))
5. **Prompt bloat tradeoff**: `pi-subagents` named-skill preloading injects full skill bodies into the system prompt. `gentle-pi` avoids that by pushing exact path reading into the child task flow. For `pi-workflow`, exact path injection is the better fit for a thin orchestrator and fresh child contexts. ([src/prompts.ts#L46-L56](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/prompts.ts#L46-L56), [extensions/skill-registry.ts#L283-L300](https://github.com/Gentleman-Programming/gentle-pi/blob/18699a40bd2cc778186516f97aa0746cf1cdeeba/extensions/skill-registry.ts#L283-L300))
6. **Silent model fallback risk in pi-subagents**: unresolved config-pinned models silently inherit the parent model, while caller-supplied unresolved/out-of-scope models error. If `pi-workflow` wants deterministic per-agent OpenAI pinning, it should validate pins proactively before launch. ([src/index.ts#L990-L1029](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/src/index.ts#L990-L1029), [README.md#L218-L231](https://github.com/tintinweb/pi-subagents/blob/8405e556acc8ab2c0fcfa02021f871365695f776/README.md#L218-L231))

## Bottom line

Use `gentle-pi` as a REFERENCE for orchestration contracts, not as a drop-in implementation blueprint.

For `pi-workflow`, the right port is:
- copy the thin orchestrator,
- copy exact-path skill injection,
- copy artifact references + read-back gatekeeping,
- keep fresh subagent context,
- pin models in agent definitions you control,
- use Engram as the durable state backend,
- and implement the skill resolver as real code instead of trusting prompt prose.

Do NOT copy the shallow registry scan, mixed `skill_resolution` enums, or the unverified `model_profiles` coupling to `@tintinweb/pi-subagents`.
